# iOS-Port M4.3 — Quest-Pin-Tap → Selektion + Highlight (Design)

**Datum:** 2026-06-29 · **Status:** entschieden (maintainer-POV, autonom freigegeben) · **Branch-Basis:** `master` (`f01380a2f`, re-synct auf `upstream/compose-quest-form`)

## Ziel & Leitlinie

Die iOS-Karte zeigt seit M4.0–M4.2 echte Quest-Pins. M4.3 macht sie **auswählbar** — als **getreue Übersetzung dessen, was Android tut**, nicht als bewusst abgespeckte „display-only"-Variante. Vorbild ist Androids `MainActivity.showQuestDetails()`-Sequenz (`androidMain/.../MainActivity.kt:964`, Schritte 2–6): Tap → Quest laden → gewählten Pin highlighten → Geometrie umranden + Kamera fokussieren → andere Pins ausblenden. Cluster-Tap zoomt rein, Tap-daneben deselektiert.

**Die eine harte Grenze:** Der Quest-**Antwort**-Formular-Pfad ist upstream-WIP. `OsmQuestFormContainer` *kann* zwar read-only rendern, aber **alle Commit-Handler sind `TODO()`** und die Upload-Pipeline fehlt. Dieser Teil wird **nicht** herbeigebaut; er bleibt eine saubere Naht.

**Bewusste Entscheidung — kein read-only-Formular in M4.3.** Android zeigt nie ein read-only-Formular; ein getippter Quest öffnet immer ein *beantwortbares* Formular. Ein read-only-Subset wäre also unfaithful und müsste, sobald das Formular kommt, wieder gelöscht werden. M4.3 endet daher bei voller Selektion + Highlight + Fokus.

## Bestätigte Fakten (Investigation `wf_a2027c0c-791`)

- **`MaplibreMap`** hat einen `onMapClick: (Position, DpOffset) -> ClickResult`-Slot (aktuell in `Map.kt` ungenutzt). `ClickResult` ist ein Enum `{ Consume, Pass }`.
- **`CameraState.animateTo(BoundingBox, bearing, tilt, PaddingValues, duration)`** existiert nativ → bbox-+-Padding-Kamerafokus ist direkt möglich (Androids inset-aware `startFocus` übersetzt sauber).
- **Keine Cluster-Leaf-API** (`getClusterLeaves`/`querySourceFeatures` fehlen auf `GeoJsonSource`) → Cluster-Expansion muss per Fallback-Zoom gelöst werden.
- `VisibleQuestsSource.get(questKey): Quest?` existiert (`VisibleQuestsSource.kt:153`); `Quest` liefert `key`, `markerLocations`, `geometry`, `type` (`type.icon` via `EditType`).
- `MapViewModel.getQuestKey(properties: Map<String,String>)` ist **bereits** vorhanden (`MapViewModel.kt:172`, delegiert an `QuestPinsManager.getQuestKey`).
- `MapViewModel` hat `setFocusedGeometry`, `setFollowing`, `viewModelScope` und `visibleQuestsSource` (Konstruktor-Arg 6) — **keine Koin-/Konstruktor-Änderung nötig**.
- `MapScreen` besitzt `cameraState` (`rememberCameraState`, `MapScreen.kt:52`) und speist `Map(...)` (`:149`). `SelectedPinsLayer(icon, pinPositions)` (`SelectedPinsLayer.kt:33`) und `FocusedGeometryLayers` sind gebaut; `FocusedGeometryLayers` ist in `Map.kt:67` schon verdrahtet, `SelectedPinsLayer` noch **nicht**. `PinsLayers(pins)` wird in `Map.kt:68` **ohne** die existierenden `onClickPin`/`onClickCluster`-Parameter aufgerufen.

## Lieferumfang (volle Android-Selektionsmaschinerie, ohne Formular)

1. **Tap → Pin auswählen.** `onClickPin` (Param existiert, `PinsLayers.kt:49`/`:124`) verdrahten.
2. **QuestKey dekodieren.** `feature.properties` (`JsonObject`) → `Map<String,String>` → `viewModel.getQuestKey(...)` (`MapViewModel.kt:172`, schon da).
3. **Quest laden.** `visibleQuestsSource.get(questKey)` off-main-thread auf `viewModelScope`.
4. **Gewählten Pin highlighten.** `SelectedPinsLayer(icon, markerLocations)` in `Map.kt` `aboveLabelsContent` als **oberste** Ebene einbinden, gespeist aus `quest.type.icon` + `quest.markerLocations`. = Androids `highlightPins()` (`SelectedPinsMapComponent.kt:71`); Multi-Location; Bounce 0.5→1.5 (iOS-Layer repliziert den OvershootInterpolator bereits via `Spring.DampingRatioMediumBouncy`).
5. **Geometrie umranden.** `FocusedGeometryLayers(geometry)` (schon verdrahtet) aus der Selektion speisen. = Androids `highlightGeometry()`.
6. **Kamera auf Geometrie fokussieren.** `cameraState.animateTo(geometry.bounds, bearing=0, tilt=0, padding=focusPadding, duration≈max(450ms, …))`; Zoom via bbox-Fit ~19 gekappt. Vor-Fokus-Kamera **einmal pro Selektions-Kette** merken (für Restore). = Androids `startFocus()`/`previousCameraPosition` (`FocusGeometryMapComponent.kt:126,151`).
7. **Andere Pins ausblenden.** `PinsLayers` bei `selectedQuest != null` verstecken (Hard-Hide, billig, wie Androids `setVisible(false)`, `MainMapFragment.kt:448`) — die Selected-Ebene bleibt oben.
8. **Cluster-Tap → reinzoomen.** `onClickCluster` (Param existiert, `PinsLayers.kt:50`/`:82`) verdrahten. **Fallback** (keine Leaf-API): auf den Cluster-Punkt zu `zoom = min(now+2, 15.0)` (> `CLUSTER_MAX_ZOOM`=14, der Filter `PinsLayers.kt:106` löst Cluster dort auf) zoomen → Cluster zerfällt in Einzel-Pins. Sichtbares Ergebnis wie Android, Framing approximativ — als bewusste Divergenz dokumentiert.
9. **Deselect.** Tap auf leere Karte (`onMapClick`) und Reselect eines anderen Pins → Selektion leeren, Pins zurück, Selected-/Geometrie-Ebenen weg, Kamera-Restore. = Androids `clearHighlighting()`+`endFocus()`.

## Architektur (Entscheidungen)

### Selektions-State im `MapViewModel`, als ein `@Stable`-Holder

iOS hat keinen `MainActivity`-Hub; der Einstieg ist `MainViewController → MainMenuNavHost → MapScreen`, das `MapViewModel` via `koinViewModel()` injiziert. `MapViewModel` besitzt bereits die Peer-Visual-States (`pins`, `geometryMarkers`, `focusedGeometry`). Selektion ist ein Peer-Belang und gehört dorthin (spiegelt `EditHistoryViewModel.selectedEdit`).

- **Verworfen:** ein separates `MainViewModel` (existiert auf iOS nicht; unbegründeter Scope) und lose parallele StateFlows (Desync-Gefahr).

```kotlin
@Stable
data class QuestSelection(
    val questKey: QuestKey,
    val icon: String,                  // quest.type.icon → Pin-Image-Name (gleicher Resolver wie Pin.icon!)
    val markerLocations: List<LatLon>, // für SelectedPinsLayer
    val geometry: ElementGeometry,     // für FocusedGeometryLayers + Kamerafokus
)
```

`MapViewModel` neu:
```kotlin
val selectedQuest: StateFlow<QuestSelection?>   // null = nichts gewählt
fun selectQuest(questKey: QuestKey)             // lädt via visibleQuestsSource.get, baut Holder, setzt State
fun clearSelection()
```

`selectQuest`: lädt den `Quest` auf `viewModelScope` mit `Dispatchers.IO` (DB-Read; konsistent mit `QuestPinsManager`/`reloadDownloadedTiles`), **no-op wenn gleicher Key** (verhindert Re-Bounce/Re-Fokus bei Klick auf bereits gewählten Pin), und dropt Follow-Mode (`setFollowing(false)`), damit die Follow-Animation den Fokus nicht überfährt. **Load-bearing:** `quest.type.icon` (`DrawableResource`) muss mit **demselben** Resolver zum Pin-Image-Namen aufgelöst werden, den `QuestPinsManager` beim Bauen von `Pin.icon` nutzt — sonst findet `SelectedPinsLayer`s `image(feature["icon-image"])` das registrierte Bild nicht. Diesen Helfer extrahieren/wiederverwenden und den Round-Trip durch `PinImageRegistry` verifizieren.

### `MapScreen` wird die Orchestrierungs-Schicht (gerechtfertigter Umbau)

Was `MainActivity` auf Android ist. Jetzt einführen: Tap-Handler, Deselect-Handler, **Kamera-Fokus/Restore-Lifecycle** und das Selektions-Overlay. **Nicht** jetzt: `Scaffold`/`ModalBottomSheet` — das ist die Naht. (Alternative — Selektion ohne Fokus-Lifecycle anflanschen — müsste beim Sheet eh neu geschrieben werden.)

## Datenfluss

```
Tap auf Pin
 → MaplibreMap routet zu PinsLayers "pins-layer" onClick = onClickPin
 → onClickPin(features: List<Feature<Geometry, JsonObject?>>)        [FeaturesClickHandler = (Feats) -> ClickResult]
 → erstes Feature; feature.properties (JsonObject) → Map<String,String> (JsonPrimitive.content; Nicht-Strings ignorieren)
 → viewModel.getQuestKey(props): QuestKey?                           [MapViewModel.kt:172]
 → questKey != null ? { viewModel.selectQuest(questKey); ClickResult.Consume } : ClickResult.Pass
 → selectQuest: quest = visibleQuestsSource.get(questKey)            [VisibleQuestsSource.kt:153]
      quest == null → clearSelection(); return
      icon = pinIconName(quest.type.icon)   // gleicher Resolver wie Pin.icon
      _selectedQuest.value = QuestSelection(questKey, icon, quest.markerLocations.toList(), quest.geometry)
      if (isFollowing) setFollowing(false)
 → MapScreen rekomponiert:
      SelectedPinsLayer(icon, markerLocations)         // Highlight (springy)
      FocusedGeometryLayers(geometry)                  // atmender Umriss
      PinsLayers ausgeblendet                          // andere Pins weg
      LaunchedEffect(selectedQuest):
         savedCamera = cameraState.position            // einmal pro Selektions-Kette
         cameraState.animateTo(geometry.bounds, bearing=0, tilt=0, padding=focusPadding, duration=…)

Cluster-Tap: onClickCluster(features) → kein Leaf-API → Fallback: animateTo(clusterPunkt, zoom=min(now+2,15)) → Consume

Deselect (→ clearSelection() + Kamera-Restore):
 - Map-Tap daneben: onMapClick = { _,_ -> if (selectedQuest != null) { clearSelection(); Consume } else Pass }
   (Pin/Cluster-Handler Consume’n, daher fällt ein Pin-Tap nie auf onMapClick durch.)
 - Reselect: selectQuest(neuerKey) ersetzt den Holder; savedCamera NUR bei der ersten Selektion der Kette merken,
   nicht bei Reselect überschreiben (Deselect kehrt zum echten Ursprung zurück).
```

`clearSelection`: Holder→null → `SelectedPinsLayer`/`FocusedGeometryLayers` verlassen die Composition (Sources via maplibre-compose-`remember`-Teardown geleert), `PinsLayers` wieder sichtbar, `LaunchedEffect` stellt `savedCamera` wieder her.

## iOS-Reconciliations & Risiken

- **ClickResult-Disziplin:** Pin/Cluster-Handler `Consume`n bei Treffer (sonst propagiert der Klick zu `onMapClick`=Deselect). `onMapClick` nur `Consume` wenn es wirklich deselektiert, sonst `Pass`. Leere/fremde Feature-Liste → `Pass`. Topmost-Hit-Test: die Selected-Ebene (oben) könnte den Klick auf den gewählten Pin abfangen — harmlos, da `selectQuest` bei gleichem Key no-op ist.
- **Cluster-Fallback** (keine `getClusterLeaves`-API): wie oben — Zoom-in statt Leaf-bbox-Fit; als bewusste Divergenz dokumentieren (Upstream-Leaf-API abwarten).
- **Kamera-Persist vs. Selektions-Fokus (subtil, gefunden in `MapScreen.kt:71`):** der Persist-Effekt speichert bei **jedem** Nicht-Follow-Move-Ende. Ein Selektions-Fokus (Follow ist aus) würde sonst die gemerkte Karten-Position mit der Quest-Position überschreiben. **Fix:** den Persist-Effekt zusätzlich auf `selectedQuest == null` gaten (analog zum bestehenden `isFollowing`-Gate). Damit überlebt die Nutzer-Kartenposition Selektion + Deselect.
- **Kamera-Padding fürs künftige Sheet:** `focusPadding` jetzt als Param (`PaddingValues(bottom=0.dp)`); wenn das Formular-Sheet kommt, `PaddingValues(bottom = sheetHeight)` durchreichen — Androids `getQuestFormInsets()`. Null Rework.
- **K/N-Threading:** maplibre-compose ruft `onClick`/`onMapClick` auf dem Main-Thread (Compose-UI). Der Handler darf den DB-Read **nicht** blockieren — er enqueued nur (`viewModelScope.launch`) und gibt `ClickResult` synchron zurück; der State wird nach dem Laden gesetzt. Alle übergebenen Typen (`LatLon`/`ElementGeometry`/`QuestKey`) sind immutable Value-Types → safe.
- **Negativ-Padding-Klemmen-Regel:** jeder neue Icon-Layer/`PaddingValues.Absolute` muss ≥0 klemmen (0.13.0 wirft zur **Laufzeit**, nicht beim Linken — siehe Memory `maplibre-compose-porting-gotchas`). `iconOffset` mit negativen Werten ist ok (Offset ≠ Padding).
- **Recompose/Perf:** `SelectedPinsLayer`/`FocusedGeometryLayers` betreten die Composition nur bei `selectedQuest != null` und re-sourcen nur bei Wertänderung (auf Wert gekeyt). Ein `@Stable`-Holder + ein StateFlow ⇒ eine Rekomposition pro Selektionswechsel. Die Haupt-Pins-Source **nicht** bei Selektion neu bauen — nur Sichtbarkeit togglen.
